Step 02 · Enrich

Raw text is just noise. Enrichment is most of the work.

A TripAdvisor review and a Reddit comment can be saying the same thing in different voices. The pipeline's job is to recognise that — and to know who said it, how strongly, and what about.

Before

"Honestly the Mona Lisa was such a letdown, we waited 20 min just to take a blurry photo from 3 metres away. Skip it and head to the Winged Victory instead — way less crowded."

One TripAdvisor row. Useful, but unstructured. We can't query "show me wait-time complaints" or "show me Winged Victory recommendations" without parsing it.

Step 02 · Enrich, in four passes

Extract, cluster, score, label.

Each pass is dumb on its own; together they turn ten thousand opinion-rows into a table you can actually query.

PASS 01
Extract
Parse the raw row into an insight skeleton — text body, rating if present, source metadata.
Each source has its own extractor because the shape differs: review fields ≠ transcript chunk fields ≠ Reddit thread fields.
OUTflat_insights_*.json
PASS 02
Cluster
Embed every insight, group semantically similar ones, name the canonical observation.
A 50-line TripAdvisor rant and a one-line Reddit comment can land in the same cluster — that's the whole point.
OUT1,450 named clusters
PASS 03
Score
Each cluster gets a quality tier — A, B, or C — based on size, source diversity, and sentiment coherence.
A claim corroborated across Reddit and TripAdvisor weighs more than the same claim repeated by twenty TripAdvisor reviews.
OUTquality_tier + source_mix
PASS 04
Label
Tag taxonomy (L1/L2/L3), intent (recommendation, warning, history…), sub-attraction, numeric anchors.
This is the pass that lets a downstream filter say "show me positive recommendations about the Mona Lisa that mention a specific time".
OUT22 fields per row

06 / 08
The interface layer

MCP exposes the table as six small tools.

Any agent or app — ours or someone else's — can plug into the same surface without re-implementing retrieval. Filter by structure, search by meaning, ask for the receipts.

search_clusters
Semantic cluster search with intent_l1 + quality_tier filters. ~200 ms.
search_insights
Per-row search with structured filters — sub_attraction, intent_tag, sentiment, has_numeric.
explain_cluster
Pulls real reviewer quotes for a cluster — the citation trail.
poi_stats
Aggregate rollups — tier / sentiment / intent counts, verdict score.
sub_attractions
Named sub-attractions with median time spent, top intent tags, source diversity.
ask
Full RAG round-trip with cited answer — the agent loop, packaged.
07 / 08
The agent · ask_gems

If we can't cite it, we don't say it.

The retrieval-augmented agent is small on purpose. It calls the MCP tools, reads the evidence, then synthesises — under a hard rule that every claim must trace to at least two source quotes.

01
Query
User asks a natural-language question about a POI.
02
Retrieve
Agent calls search_clusters / search_insights — filtered, not just semantic.
03
Cite
explain_cluster pulls the raw reviewer quotes for the candidate clusters.
04
Answer
Structured output — answer + ≥2 citations + confidence — never free prose.
The discipline: if retrieval returns fewer than two corroborating quotes, the agent sets confidence to "low" and admits the thinness in the answer. We chose this over hedging or polite fabrication.
